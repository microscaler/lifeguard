# Chicken-and-Egg Dependency Analysis: LifeQuery vs LifeModel

**Version:** 1.0  
**Date:** 2025-01-XX  
**Purpose:** Analyze and resolve potential circular dependency between Epic 02 (LifeQuery) and Epic 03 (LifeModel)

---

## The Potential Problem

### Question
Does LifeQuery need LifeModel's column enums to work?  
Does LifeModel need LifeQuery to work?

### The Concern
Looking at SeaORM usage:
```rust
User::find()
    .filter(User::Column::Email.eq("test@example.com"))
    .all(&pool)?;
```

The `User::Column::Email` is generated by the `#[derive(LifeModel)]` macro.  
But LifeQuery needs to know about columns to build queries.  
But LifeModel needs LifeQuery to build queries.

**Is this circular?**

---

## Analysis: How SeaQuery Actually Works

### SeaQuery is Generic

SeaQuery itself is **runtime-agnostic** and **model-agnostic**. It works with:

1. **String-based column references:**
```rust
use sea_query::{ColumnRef, Alias, PostgresQueryBuilder};

let query = SelectStatement::new()
    .column(ColumnRef::new("email"))
    .from(Alias::new("users"))
    .and_where(ColumnRef::new("email").eq("test@example.com"))
    .to_string(PostgresQueryBuilder);
```

2. **Alias-based table references:**
```rust
let table = Alias::new("users");
let query = SelectStatement::new()
    .from(table)
    .to_string(PostgresQueryBuilder);
```

### SeaORM Adds Type Safety

SeaORM's `DeriveEntityModel` macro generates:
- `Column` enum: `User::Column::Email`, `User::Column::Id`, etc.
- These enums **wrap** SeaQuery's generic `ColumnRef`
- They provide **compile-time type safety** but aren't required for SeaQuery to work

---

## Solution: Two-Phase LifeQuery Implementation

### Phase 1: Generic LifeQuery (Epic 02)

**Build LifeQuery as a generic SQL builder** that works with:
- String-based column references: `"email"`, `"id"`, etc.
- Alias-based table references: `Alias::new("users")`
- Direct SeaQuery statements

**Example API:**
```rust
// Generic LifeQuery - works without LifeModel
let query = LifeQuery::select()
    .column("email")
    .from("users")
    .filter("email", "eq", "test@example.com")
    .build();

let rows = pool.execute_query(query)?;
```

**This is testable independently:**
- Can test SQL generation
- Can test parameter binding
- Can test execution via `may_postgres`
- **No LifeModel required**

---

### Phase 2: Type-Safe LifeQuery Integration (Epic 03)

**Enhance LifeQuery to work with LifeModel's column enums:**

```rust
// Type-safe LifeQuery - uses LifeModel's column enums
impl User {
    pub fn find() -> LifeQueryBuilder<Self> {
        LifeQueryBuilder::new(Self::TABLE)
            .with_columns(User::Column::as_refs()) // Generated by macro
    }
}

// Usage
User::find()
    .filter(User::Column::Email.eq("test@example.com")) // Type-safe!
    .all(&pool)?;
```

**The column enum wraps the generic column reference:**
```rust
// Generated by LifeModel macro
impl User::Column {
    fn as_column_ref(&self) -> ColumnRef {
        match self {
            User::Column::Email => ColumnRef::new("email"),
            User::Column::Id => ColumnRef::new("id"),
            // ...
        }
    }
}
```

---

## Dependency Resolution

### ✅ No Circular Dependency

**Epic 02 (LifeQuery):**
- Builds **generic** SQL builder
- Works with **string-based** column references
- Executes via `may_postgres`
- **Independent of LifeModel**

**Epic 03 (LifeModel):**
- Generates **type-safe** column enums
- These enums **wrap** LifeQuery's generic column references
- Provides **convenience layer** on top of LifeQuery
- **Depends on LifeQuery** (but LifeQuery doesn't depend on it)

### The Flow

```
Epic 02: LifeQuery (Generic)
    ↓
Epic 03: LifeModel (Type-Safe Wrapper)
    ↓
User::Column::Email → wraps → LifeQuery::column("email")
```

    assert_eq!(

        "SELECT id, email FROM users WHERE email = $1"

## Key Insight: Wrapper Pattern

### LifeQuery is the Foundation

LifeQuery provides:
- **Generic SQL building** (works with strings)
- **Execution via may_postgres**
- **Parameter binding**
- **All query operations**

### LifeModel is the Enhancement

LifeModel provides:
- **Type-safe column references** (wraps LifeQuery's strings)
- **Convenience methods** (`find()`, `find_by_id()`, etc.)
- **Row-to-struct mapping**
- **Compile-time safety**

### The Relationship

```
LifeQuery (Generic)
    ↑ wraps
LifeModel (Type-Safe)
```

**Not circular - it's a wrapper pattern!**

## Verification: No Circular Dependency

### Can Epic 02 be built without Epic 03?

**YES** ✅
- LifeQuery uses generic string-based column references
- Can test SQL generation independently
- Can test execution with raw SQL
- No LifeModel required

### Can Epic 03 be built without Epic 02?

**NO** ❌
- LifeModel needs LifeQuery to build SQL
- Column enums wrap LifeQuery's `ColumnRef`
- Query builders use LifeQuery internally

### Conclusion

**No circular dependency!**  
LifeQuery is the foundation (generic).  
LifeModel is the enhancement (type-safe wrapper).

## Final Answer

### ❌ No Chicken-and-Egg Problem

**Reason:**
1. LifeQuery is **generic** (works with strings)
2. LifeModel is **type-safe wrapper** (wraps LifeQuery's strings)
3. LifeQuery can be built and tested **independently**
4. LifeModel **depends on** LifeQuery, but LifeQuery doesn't depend on LifeModel

**The Pattern:**
```
Generic Foundation (LifeQuery)
    ↑ wrapped by
Type-Safe Enhancement (LifeModel)
```

This is a **standard wrapper pattern**, not a circular dependency.
