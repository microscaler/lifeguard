# BUG-2026-01-21-04: Migration Version Parsing Error Loses Diagnostic Information

**Date:** 2026-01-21  
**Status:** ✅ FIXED  
**Priority:** High  
**Severity:** Diagnostic information loss - Users see misleading error messages

## Summary

When parsing a migration version fails, the error handler attempts to parse the same string again and uses `unwrap_or(0)`, resulting in `MigrationError::InvalidVersion(0)` instead of preserving the actual invalid value. This loses diagnostic information - users see "Invalid migration version: 0" even though the actual problematic version string was something else entirely.

## Discovery

**Date:** 2026-01-21  
**Source:** Code review / User report  
**Severity:** `high`  
**Status:** `fixed`

## Location

- **File:** `src/migration/file.rs`
- **Lines:** 50-53 (original buggy code)

## Description

The `parse_filename` method in `MigrationFile` has a bug in its error handling:

```rust
let version = version_str.parse::<i64>()
    .map_err(|_e| MigrationError::InvalidVersion(
        version_str.parse::<i64>().unwrap_or(0)
    ))?;
```

**The Problem:**
1. `version_str.parse::<i64>()` fails (returns an error)
2. The error handler tries to parse `version_str` again with `version_str.parse::<i64>()`
3. This second parse will also fail (same string, same problem)
4. So `unwrap_or(0)` is used, resulting in `0` being passed to `InvalidVersion`
5. The actual `version_str` value is lost

**Example:**
- If a migration file is named `m2024012012000X_create_table.rs` (invalid character 'X')
- The error message shows: "Invalid migration version: 0"
- The user has no idea what the actual invalid value was

## Root Cause

The error handling code attempts to parse the string again in the error handler, which will always fail for the same reason. The `unwrap_or(0)` fallback masks the original invalid value.

Additionally, `MigrationError::InvalidVersion` was defined to accept an `i64`, which doesn't make sense when the value couldn't be parsed as an integer in the first place.

## Fix

1. **Changed `InvalidVersion` variant** to accept a `String` instead of `i64`:
   - Updated `src/migration/error.rs` to change `InvalidVersion(i64)` to `InvalidVersion(String)`
   - Updated the `Display` implementation to show the original invalid string with helpful context

2. **Fixed the parsing code** to preserve the original string:
   ```rust
   let version = version_str.parse::<i64>()
       .map_err(|_e| MigrationError::InvalidVersion(
           version_str.to_string()
       ))?;
   ```

3. **Updated other usage** in `lifeguard-migrate/src/main.rs`:
   - Changed from `InvalidVersion(version)` to `InvalidVersion(version.to_string())`
   - This case is different (valid i64 that doesn't exist), but converting to string maintains consistency

## Testing

The fix preserves diagnostic information:
- Invalid version strings are now shown in error messages
- Users can see exactly what value failed to parse
- Error message includes format hint: "expected format: YYYYMMDDHHMMSS"

## Impact

- **User Experience:** Error messages now show the actual invalid value instead of "0"
- **Debugging:** Developers can immediately see what went wrong
- **Diagnostics:** Preserves all diagnostic information for troubleshooting
