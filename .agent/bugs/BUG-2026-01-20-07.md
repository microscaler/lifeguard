# BUG-2026-01-20-07: Migration Test Hanging on Lock Acquisition

**Date:** 2026-01-20  
**Status:** 🔴 OPEN  
**Priority:** Critical  
**Severity:** Test hangs indefinitely - blocks CI/CD

## Summary

The `test_migration_lifecycle` integration test hangs indefinitely (5400+ seconds) when trying to acquire a migration lock. The timeout mechanism in `acquire_migration_lock` only checks timeout AFTER database queries complete, so if a database query hangs, the timeout never triggers.

## Discovery

**Date:** 2026-01-20  
**Source:** Test execution - test running for 5400+ seconds  
**Severity:** `critical`  
**Status:** `open`

## Location

- **File:** `src/migration/lock.rs`
- **Lines:** 85-127 (`acquire_migration_lock` function)
- **Test:** `tests-integration/migration_test.rs:294` (`test_migration_lifecycle`)

## Description

The `acquire_migration_lock` function has a timeout mechanism that checks `start.elapsed() >= timeout` after each database query attempt. However, if the database query itself hangs (e.g., due to network issues, database locks, or connection problems), the timeout check never executes because the code is blocked waiting for the query to return.

### Current Flow (Problematic)

```rust
loop {
    // This query may hang indefinitely
    let rows_affected = executor.execute(&sql, &[])?;
    
    if rows_affected > 0 {
        return Ok(());
    }
    
    // Timeout check only happens if query completes
    if start.elapsed() >= timeout {
        return Err(MigrationError::LockTimeout(...));
    }
    
    std::thread::sleep(Duration::from_millis(100));
}
```

### Problem

1. If `executor.execute()` hangs (never returns), the timeout check never runs
2. The test waits indefinitely for the query to complete
3. No way to interrupt or cancel the hanging query

## Root Cause

1. **No query timeout**: Database queries don't have a built-in timeout mechanism
2. **Timeout check placement**: Timeout is checked AFTER query completes, not during execution
3. **Blocking I/O**: The `may_postgres` executor uses blocking I/O, so hanging queries block the entire thread
4. **No cancellation mechanism**: There's no way to cancel a hanging database query

## Potential Solutions

### Option 1: Add Query Timeout (Recommended)
- Use PostgreSQL's `statement_timeout` setting to force queries to timeout
- Set timeout before each query attempt
- Reset timeout after query completes

### Option 2: Use Async Timeout Wrapper
- Wrap the query in a timeout mechanism using threads or async runtime
- Cancel the query if it exceeds the timeout
- Requires refactoring to support cancellation

### Option 3: Improve Lock Cleanup
- Ensure locks are always released in tests
- Add verification that locks are released before test starts
- Add panic if lock is held when test begins

### Option 4: Use Non-Blocking Lock Check
- Check if lock exists before trying to acquire it
- Only attempt acquisition if lock is not held
- Reduces chance of hanging on lock acquisition

## Immediate Workaround

1. **Manual lock cleanup**: If test hangs, manually delete lock:
   ```sql
   DELETE FROM lifeguard_migrations WHERE version = -1;
   ```

2. **Test improvements**: Added lock verification in test setup to panic early if lock is held

3. **Better cleanup**: Improved test cleanup to ensure locks are always released
