# BUG-2026-01-21-05: register_migration Uses Wrong Error Type for Already Registered Migrations

**Date:** 2026-01-21  
**Status:** âś… FIXED  
**Priority:** High  
**Severity:** Semantic mismatch - Error handling logic confusion

## Summary

The `register_migration` function returns `MigrationError::AlreadyApplied` when a migration with the same version is already registered in the in-memory registry. However, `AlreadyApplied` semantically means the migration was already applied to the database, not that it's registered in memory. This mismatch could confuse error handling logic that expects `AlreadyApplied` to indicate database state.

## Discovery

**Date:** 2026-01-21  
**Source:** Code review / User report  
**Severity:** `high`  
**Status:** `fixed`

## Location

- **File:** `src/migration/registry.rs`
- **Lines:** 37-39 (original buggy code)

## Description

The `register_migration` function checks if a migration with the same version is already registered in the in-memory registry:

```rust
if registry.contains_key(&version) {
    return Err(MigrationError::AlreadyApplied { version, name });
}
```

**The Problem:**
1. `AlreadyApplied` semantically means "migration was already applied to the database"
2. The error message confirms this: "Migration '{}' (version {}) has already been applied"
3. However, `register_migration` is checking if a migration is already registered in memory (the registry HashMap)
4. These are two different states:
   - **Registered in memory**: Migration is loaded into the runtime registry
   - **Applied to database**: Migration has been executed and recorded in the database

**Impact:**
- Error handling code that checks for `AlreadyApplied` expects it to mean database state
- This could lead to incorrect logic when handling registration errors
- The semantic mismatch makes the codebase harder to understand and maintain

## Root Cause

The error type `AlreadyApplied` was incorrectly used for a different semantic meaning. The function should use a distinct error type to indicate that a migration is already registered in memory, separate from whether it's been applied to the database.

## Fix

1. **Added new error variant** `AlreadyRegistered` to `MigrationError` enum:
   - Added in `src/migration/error.rs` after `AlreadyApplied`
   - Includes `version` and `name` fields for consistency with `AlreadyApplied`

2. **Updated Display implementation** to provide clear error message:
   ```rust
   MigrationError::AlreadyRegistered { version, name } => {
       write!(
           f,
           "Migration '{}' (version {}) is already registered in the migration registry",
           name, version
       )
   }
   ```

3. **Fixed `register_migration`** to use the correct error type:
   ```rust
   if registry.contains_key(&version) {
       return Err(MigrationError::AlreadyRegistered { version, name });
   }
   ```
